#### 步骤 6：运行 DQ Pipeline

```powershell
docker compose run --rm spark-dq-app python3 src/main_pipeline.py
```

**核心流程（7 步）：**

```
① load_config()
   └─ 解析 config/dq_checks_config.yaml → Python dict

② get_spark_session("DataQualityPipeline")
   └─ SparkSession.builder.master("local[*]")
   └─ spark.driver.memory = 2g
   └─ spark.executor.memory = 2g

③ spark.read.parquet(path)
   └─ 加载 synthetic_data.parquet → df_raw

④ apply_data_cleaning(df_raw)
   ├─ price < 0 → 0.0
   ├─ 无效 email → NULL
   ├─ 非法 country_code → "UNKNOWN"
   ├─ 非法 order_status → "UNKNOWN"
   ├─ 日期颠倒 → dates_marked_inconsistent = True
   └─ Window去重:
        ROW_NUMBER() OVER (PARTITION BY transaction_id ORDER BY start_date DESC)
        保留 row_num=1 (最新记录)
        重要: null key行分离处理，避免全部null分到同一分区被误删

⑤ df_cleaned.cache()
   └─ 缓存清洗后的数据到内存 (避免17条检查重复计算)

⑥ for each check in config (18条):
   ├─ get_check_function(check_type, sub_type)  # 策略模式路由
   ├─ result = check_function(df, **params)      # 执行检查
   └─ results[check_name] = result               # 收集到字典

⑦ write_report(results) + log_to_postgres(results)
   └─ JSON + CSV 报告 → reports/
   └─ INSERT INTO dq_results_log → PostgreSQL
```

## 四、18 条 DQ 规则详解

### 4.1 完整性检查 (3 条)

| 检查名 | 列 | 阈值 | 注入错误率 | 逻辑 |
|--------|-----|------|-----------|------|
| check_user_id_not_null | user_id | 0% | 0% | `isNull() \| isnan()` 计数 |
| check_price_not_null | price | 6% | 5% | NaN 和 NULL 分别检测 |
| check_quantity_not_null | quantity | 6% | 5% | 同上 |

**关键细节：** Spark 中 `NaN ≠ NULL`。`F.isnull()` 查 SQL NULL，`F.isnan()` 查 float NaN。两个条件必须用 `|` 组合，否则会漏检。

### 4.2 唯一性检查 (2 条)

| 检查名 | 列 | 阈值 | 注入错误率 |
|--------|-----|------|-----------|
| check_user_id_unique | user_id | 0% | 0% |
| check_transaction_id_unique_post_dedupe | transaction_id | 0.1% | 3% (去重后 0%) |

**实现算法：**
```
non_null_df.groupBy(column).count()
  → filter(count > 1)                        # 找到重复值
  → inner join 回原表                          # 计数重复行数
  → duplicate_ratio = 重复行数 / 总行数
```

**关键细节：** NULL 值在 SQL 标准中不视为重复。实现中先 `filter(isNotNull())` 再查重，全 NULL 列返回 PASS。

### 4.3 有效性检查 (4 条)

| 检查名 | 子类型 | 参数 | 阈值 |
|--------|--------|------|------|
| check_price_range_post_clean | range | min=0.0 | 0.1% |
| check_email_format_post_clean | regex | 邮箱正则 | 0.1% |
| check_country_code_values_post_clean | categorical | 7 个白名单值 | 0.1% |
| check_order_status_values_post_clean | categorical | 5 个白名单值 | 0.1% |

**实现模式：**
```python
# range: 动态组合条件
conditions = [col < min] + [col > max]
df.filter(isNotNull() & (cond1 | cond2)).count()

# regex: 反向匹配
df.filter(isNotNull() & ~col.rlike(pattern)).count()

# categorical: 不在白名单
df.filter(isNotNull() & ~col.isin(allowed_values)).count()
```

**关键细节：** 这 4 条检查名为 `*_post_clean`，运行在清洗**之后**的数据上。清洗步骤已将无效值转为 UNKNOWN/NULL/0.0，所以理论上这 4 条应全部返回 0% 异常——实际运行结果也确实验证了清洗逻辑的正确性。

### 4.4 一致性检查 (2 条)

| 检查名 | 子类型 | 参数 | 阈值 |
|--------|--------|------|------|
| check_end_date_after_start_date_final | custom_condition | `end_date >= start_date` | 12% |
| check_user_id_exists_in_users_ref | referential_integrity | users.parquet | 3% |

**custom_condition 实现：**
```python
df.filter(F.expr(f"NOT ({condition_sql})")).count()
```
支持任意 SQL 表达式，YAML 写什么就检查什么。

**referential_integrity 实现（面试亮点）：**
```python
# left_anti join = Spark的 "NOT EXISTS" 优化
ref_keys = spark.read.parquet(ref_path).select(ref_col).distinct()
missing = df.join(ref_keys, df[col] == ref_keys[ref_col], how='left_anti')
missing.count()
```

`left_anti` 是 Spark SQL 对 `NOT EXISTS` 子查询的专门优化——只保留左表有但右表没有的行，不需要物化 join 结果。1M 行 vs 985K 引用表的规模下约 2-3 秒完成。

### 4.5 格式检查 (6 条)

| 检查名 | 子类型 | 检查内容 |
|--------|--------|----------|
| check_product_id_length | string_length | 固定长度 = 4 |
| check_country_code_length_post_clean | string_length | 长度 2~7 |
| check_user_id_data_type | data_type | bigint |
| check_price_data_type_post_clean | data_type | double |
| check_start_date_data_type | data_type | timestamp |
| check_dates_marked_inconsistent_data_type | data_type | boolean |

**data_type 检查实现：**
```python
actual_type_str = df.schema[column].dataType.simpleString()  # "bigint"
is_match = actual_type_str == expected_spark_type_str.strip().lower()
```

### 4.6 统计检查 (1 条)

| 检查名 | 子类型 | 参数 |
|--------|--------|------|
| check_price_mean_deviation | mean_deviation | expected_mean=255.0, max_deviation=3σ |

**实现（Z-score 方法）：**
```python
stats = df.select(F.mean(col), F.stddev_pop(col)).first()
deviation = abs(actual_mean - expected_mean) / actual_stddev
status = "FAIL" if deviation > max_deviation_stddevs else "PASS"
```

### Bug 3：pip 权限错误

**报错：**
```
ERROR: Could not install packages due to an OSError:
[Errno 13] Permission denied: '/.local'
```

**原因：** `apache/spark-py` 基础镜像中默认用户 `spark`（uid 185）没有写入 `/.local` 的权限，pip 默认使用 user installation 目录。

**解决：** 在 `RUN pip install` 之前加 `USER root`，安装完成后切回 `spark` 用户。后因镜像中无 spark 用户，简化为全程 root。

**修改文件：** `Dockerfile`
```dockerfile

USER root
...
```

### Bug 4：pyspark 版本不匹配

**报错：**
```
TypeError: 'JavaPackage' object is not callable
```

**现象：** Pandas 数据生成成功，但 `SparkSession.builder.getOrCreate()` 失败。

**原因：** 基础镜像内置的是 Spark 3.4.0（Java 侧），但 pip 不指定版本时安装了 pyspark 4.1.2（Python 侧）。Python 客户端和 Java 服务端版本不兼容。

**解决：** 锁定 pyspark 版本匹配镜像中的 Spark 版本。

**修改文件：** `Dockerfile`
```dockerfile

```

**教训：** Docker 镜像中安装 pyspark 时，必须确保 pip 安装的版本与镜像中 `spark-submit --version` 显示的 Java 端 Spark 版本一致。

### Bug 5：stats 检查返回 NaN

**报错 / 现象：**
```
check_price_mean_deviation: Status=ERROR, Metric=1.0
Message: Could not calculate valid stats (mean=nan, stddev=nan)
```

**定位过程：**
1. 检查 data_generator.py → 确认 price 列有 5% 的 `np.nan` 注入
2. 检查 `apply_data_cleaning()` → 清洗步骤没有处理 NaN（只处理了 `price < 0`）
3. 检查 Spark 文档 → `F.mean()` 在列中包含 NaN 值时，结果也是 NaN
4. 验证：`F.isnan()` 和 `F.isNull()` 是不同的条件

**根因：** Spark 中 `NaN ≠ NULL`。`np.nan` 被 Spark 视为特殊的 float 值而非 NULL。`F.mean()` 在计算时包含 NaN → 均值变成 NaN → 标准差也变成 NaN → 检查返回 ERROR。

**解决：** 在统计计算前过滤 NaN 值。

**修改文件：** `src/dq_checks/stats.py`
```python
# 修复前：
stats = df.select(F.mean(col), F.stddev_pop(col)).first()

# 修复后：
clean_df = df.filter(F.col(column).isNotNull() & ~F.isnan(F.col(column)))
stats = clean_df.select(F.mean(col), F.stddev_pop(col)).first()
```

**修复后运行结果：**
```
check_price_mean_deviation: Status=PASS, Metric=0.035
actual_mean=249.98, StdDev=144.41
Expected mean=255.00 → deviation=0.035σ (阈值 3σ)
```

**面试价值：** 这是一个真实的调试故事——从发现问题→定位 NaN≠NULL 的 Spark 特性→阅读文档验证→写修复→回归验证。比"一切顺利跑通"更能展示排查能力。

## 七、核心代码片段（面试讲解用）

### 7.1 策略模式路由

```python
# src/main_pipeline.py: get_check_function()
def get_check_function(check_type, sub_type=None):
    if check_type == "completeness":
        return completeness.check_not_null
    elif check_type == "validity":
        if sub_type == "range":    return validity.check_range
        elif sub_type == "regex":  return validity.check_regex
        elif sub_type == "categorical": return validity.check_allowed_values
    elif check_type == "consistency":
        if sub_type == "referential_integrity":
            return consistency.check_referential_integrity
        ...
```
**面试话术：** "这是手工实现的策略模式——根据 YAML 配置中的 (type, subtype) 动态路由到具体检查函数。新增检查类型只需加一个 elif 分支 + 实现函数，符合开闭原则。"

### 7.2 Window 去重

```python
# src/main_pipeline.py: apply_data_cleaning()
window_spec = Window.partitionBy("transaction_id") \
                     .orderBy(F.col("start_date").desc())
df_deduplicated = df.withColumn("row_num", F.row_number().over(window_spec)) \
                    .filter(F.col("row_num") == 1) \
                    .drop("row_num")
```
**等价 SQL：**
```sql
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY transaction_id ORDER BY start_date DESC
  ) AS rn FROM transactions
) WHERE rn = 1
```
**面试话术：** "3 万行故意重复数据，用 Window 函数按最新 start_date 去重。关键是 null key 行必须分离处理——如果 transaction_id 为 NULL 的行参与 Window 分区，所有 NULL 会被放入同一个分区，只保留一行。"

### 7.3 left_anti join

```python
# src/dq_checks/consistency.py: check_referential_integrity()
ref_keys = ref_df.select(F.col(reference_column)).distinct()
missing = df.join(ref_keys,
    df[column] == ref_keys[f"ref_{reference_column}"],
    how='left_anti')
```
**面试话术：** "left_anti 是 Spark 对 NOT EXISTS 的优化——不物化 join 结果，只保留左表有右表没有的行。1M vs 985K 的规模下约 2-3 秒完成。"

### 7.4 .cache() 优化
